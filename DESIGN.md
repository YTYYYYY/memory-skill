# Memory Skill - 设计原型参考

## 1. 设计背景与目标

### 1.1 背景
AI Agent 在多会话环境中缺乏持久化记忆能力，每个会话都是独立的，无法继承之前会话的上下文。这导致：
- 用户需要重复告诉 Agent 自己的偏好
- 任务进度无法跨会话保持
- 重要决策无法追溯

### 1.2 设计目标
1. **跨会话持久化** - 数据存储在 MinIO，会话结束后不丢失
2. **Agent数据隔离** - 每个 Agent 有独立的数据空间
3. **操作可追溯** - 所有操作记录在索引中，支持回溯
4. **自动备份** - 每次操作后自动同步，无需手动备份

## 2. 核心设计

### 2.1 系统架构

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Agent     │────▶│  memory.sh  │────▶│   MinIO     │
│  (调用方)   │     │  (核心脚本)  │     │  (存储后端)  │
└─────────────┘     └─────────────┘     └─────────────┘
       │                   │
       │            ┌──────▼──────┐
       │            │  index.json │
       │            │   (索引)    │
       │            └─────────────┘
       ▼
┌─────────────┐
│  本地存储   │
│ (工作目录)  │
└─────────────┘
```

### 2.2 数据模型

#### 任务 (Task)
```json
{
  "task_id": "string",
  "phase": "string",
  "created_at": "ISO8601",

  ],
  "current_context": {"key": "value"},
  "key_decisions": [
    {"decision": "string", "why": "string", "timestamp": "ISO8601"}
  ]

```

#### 项目 (Project)
```json
{
  "project_id": "string",
  "created_at": "ISO8601",
  "background": "string",
  "goals": ["string"],
  "team": ["string"],
  "constraints": ["string"]

```

#### 索引 (Index)
```json
{
  "tasks": {
    "task_id": {
      "phase": "string",
      "updated_at": "ISO8601",
      "tags": ["string"],
      "keywords": ["string"]
    }
  },
  "projects": {}
}
```

## 3. 模块设计

### 3.1 命令行接口 (CLI)

所有操作通过 `memory.sh` 脚本完成：

| 模块 | 功能 |
|------|------|
| 模板管理 | 预定义任务模板，自定义模板 |
| 任务管理 | 创建、更新、删除任务 |
| 进度跟踪 | 保存步骤、决策、上下文 |
| 搜索检索 | 关键词、标签、时间范围搜索 |
| 备份恢复 | 增量/全量备份，自动恢复 |

### 3.2 备份机制

三种备份模式：

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| `sync` | 每次操作立即同步 | 数据安全要求高 |
| `incremental` | 只备份修改的文件 | 网络带宽有限 |
| `lazy` | 手动备份 | 开发调试阶段 |

### 3.3 版本控制

- 每次备份更新 `.version` 文件
- 支持版本校验和冲突检测
- 必要时可从 MinIO 恢复

---

## 4. 安全性设计

### 4.1 数据隔离
- 按 Agent 名称隔离数据
- MinIO 路径：`hiclaw/hiclaw-storage/agents/{AGENT_NAME}/memory/`

### 4.2 权限控制
- 只读 MinIO bucket
- 通过 MCP Server 统一管理 API 访问

---

## 5. 扩展性设计

### 5.1 自定义模板
支持添加、编辑、删除自定义模板：
```bash
memory.sh add-template mytemplate "步骤1,步骤2,步骤3"
```

### 5.2 插件化
- 可扩展的存储后端（当前支持 MinIO）
- 可扩展的索引方式

---

## 6. 设计决策记录

| 决策 | 原因 | 影响 |
|------|------|------|
| 使用 JSON 格式存储 | 易读、易解析、易版本控制 | 需要 jq 依赖 |
| 按 Agent 名称隔离数据 | 保证多租户安全 | 数据结构简单 |
| 使用 Bash 脚本 | 无需额外运行时 | 功能有一定局限性 |
| 三层备份模式 | 平衡性能和数据安全 | 用户需要理解模式差异 |
